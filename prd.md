# 02. PRD — 블로그 CRUD 프로젝트

작성일 2026-07-21 · 작성자 [김성미] · 상태 초안 v1

제품 요구사항 명세(PRD) — 개요·목표·범위·기능/비기능 요구사항·유저스토리·수용기준(Given/When/Then)·MoSCoW·마일스톤·의존성

입력: `crud-pure-clarke.md`(테이블 설계 ERD + 페이지 목록 v1). 본 문서는 "무엇을/왜"에 집중하며, "어떻게(기술 스택·구현)"는 개발 착수 단계에 위임한다. **본 PRD는 개발 착수 전에 작성되며, 이후 로컬 구현 → Supabase 연동 → 배포 순으로 진행한다.**

## 1. 개요 & 목적

**제품**: 블로그 CRUD 웹앱 (개인 단위, 회원 인증 기반)

**배경**: AI빌더스쿨 2주차 과제 — CRUD(생성/조회/수정/삭제)와 Supabase 연동을 실습하기 위한 학습 프로젝트. 화면에 보이는 것과 실제 저장(DB)의 차이, 인증·인가·해싱의 개념을 실제 동작하는 서비스로 확인하는 것이 목표.

**문제**: 데이터를 다루는 서비스를 만들려면 ① 회원 인증 ② 본인 데이터만 수정 가능한 권한 통제 ③ 비밀번호 등 민감정보의 안전한 저장 ④ 실제 DB(Supabase)와의 연동이 전부 갖춰져야 하는데, 이 네 가지가 함께 동작하는 실물이 아직 없다.

**목적**: 로그인한 회원이 글을 쓰고 자신의 글만 수정·삭제할 수 있으며, 방문자는 목록·상세를 열람할 수 있는 CRUD 흐름을 Supabase 위에 구현한다.

**핵심 가치 제안**: "가입부터 글 작성·수정·삭제까지, 내 블로그를 스스로 관리한다."

**1차 타겟**: 이번 과제를 수행하는 본인 — 실제 서비스 관점에서는 "글을 써서 관리하고 싶은 개인 사용자".

| 기능 완성도 | CRUD(생성/조회/수정/삭제) 전체 동작 여부 | 4개 기능 모두 성공 | 로컬(Live Server)에서 각 기능 수동 테스트 |
| 데이터 연동 | Supabase 실데이터 반영률 | 100% | 앱에서 조작한 결과와 Supabase 테이블 편집기 대조 |

## 3. 범위 (Scope)

### In Scope (v1)
- 회원가입/로그인 (이메일 + 비밀번호, 해시 저장)
- 게시글 CRUD: 작성(Create)·목록/상세 조회(Read)·수정(Update)·삭제(Delete)
- 카테고리·태그 분류 및 필터링
- 대표 이미지 업로드
- Supabase 연동 — 실데이터 저장/조회 확인, 게시글 관리(내 글 목록에서 수정·삭제)
- 마이페이지(내 글 목록, 프로필 수정)
- 로컬(Live Server)에서 동작 확인 후 배포

### Out of Scope (v1 제외)
- 댓글 기능 (강의에서 NoSQL 확장으로 다뤘으나 이번 과제 필수 범위 아님)
- 소셜 로그인(카카오·구글) — 이메일/비밀번호 로그인 기준
- 예약·어드민 기능 — **블로그와 무관한 별도 프로젝트**(지난 세션부터 진행한 사장님 예약 시스템)의 범위이며 이번 문서와 관련 없음
- 결제, 알림, 검색엔진 최적화(SEO), 실시간 협업

**릴리스 경계**: 단일 마일스톤(M1: 핵심 CRUD 루프)로 기한 내 전체 완성 — 별도 단계형 릴리스 없음.

## 4. 기능 요구사항 (FR)

기능 영역별로 유저스토리 + 수용기준(Given/When/Then)을 정의한다. 화면·경로는 `crud-pure-clarke.md`의 "2. 페이지 목록"을 따른다.

### FR-1 · 인증 (회원가입/로그인, Supabase Auth)
- US-1.1 방문자로서, 이메일과 비밀번호로 가입해 회원이 되고 싶다.
- US-1.2 회원으로서, 로그인해 내 글을 작성·관리하고 싶다.

- AC-1.1 Given 비로그인 사용자가 `/signup`에서 이메일/비밀번호/닉네임 입력, When 제출, Then `supabase.auth.signUp()`으로 계정이 생성되고(비밀번호는 Supabase가 자동 해시 저장) 트리거로 `profiles` 행이 함께 생성되며 로그인 상태로 전환된다.
- AC-1.2 Given 이미 가입된 이메일, When 회원가입 제출, Then Supabase Auth가 반환하는 중복 이메일 에러를 화면에 표시하고 가입을 막는다.
- AC-1.3 Given 로그인 폼에 잘못된 이메일/비밀번호 입력, When `supabase.auth.signInWithPassword()` 호출, Then 인라인 에러를 표시하고 로그인시키지 않는다.

### FR-2 · 게시글 작성 (Create)
- US-2.1 로그인한 사용자로서, 제목·본문·카테고리·태그·이미지를 입력해 글을 작성하고 싶다.

- AC-2.1 Given 로그인 상태로 `/posts/new` 진입, When 제목·본문 없이 제출, Then 필수값 에러를 표시하고 저장하지 않는다.
- AC-2.2 Given 필수값을 채운 상태, When 제출, Then Supabase `posts` 테이블에 새 행이 생성되고 상세 페이지(`/posts/:id`)로 이동한다.
- AC-2.3 Given 이미지 업로드, When 저장, Then 이미지 URL이 `posts.thumbnail_image`에 저장된다.

### FR-3 · 게시글 조회 (Read)
- US-3.1 방문자로서, 전체 글 목록을 보고 카테고리·태그로 필터링하고 싶다.
- US-3.2 방문자로서, 글 상세를 클릭해 전체 내용을 읽고 싶다.

- AC-3.1 Given 홈(`/`), When 목록 로드, Then Supabase에서 조회한 게시글이 최신순으로 표시되고 페이지네이션이 적용된다.
- AC-3.2 Given 카테고리/태그 선택, When 필터 적용, Then 해당 조건의 글만 목록에 남는다.
- AC-3.3 Given 상세 페이지 진입, When 로드, Then 제목/본문/작성자/작성일/카테고리/태그가 표시되고 `view_count`가 1 증가한다.

### FR-4 · 게시글 수정·삭제 (Update/Delete) ★ 권한 검증 핵심
- US-4.1 작성자 본인으로서, 내가 쓴 글을 수정하거나 삭제하고 싶다.
- US-4.2 사용자로서, 남의 글은 수정·삭제할 수 없어야 안심할 수 있다.

- AC-4.1 Given 작성자 본인이 `/posts/:id/edit` 접근, When 수정 제출, Then 변경사항이 Supabase에 반영된다.
- AC-4.2 Given 작성자가 아닌 로그인 사용자가 `/posts/:id/edit`에 URL로 직접 접근하거나 API를 직접 호출, When 수정/삭제 요청, Then Supabase RLS 정책(`auth.uid() = posts.user_id`)이 DB 단에서 거부한다 — 클라이언트 JS 우회로도 뚫리지 않는다.
- AC-4.3 Given 삭제 버튼 클릭, When 확인 다이얼로그에서 확인, Then 해당 행이 삭제되고 목록으로 리다이렉트된다.

### FR-5 · Supabase 연동 및 게시글 관리
- US-5.1 작성자로서, Supabase에 내 게시글 데이터가 정확히 반영되는지 확인하고 관리하고 싶다.

- AC-5.1 Given Supabase 테이블 편집기, When `posts` 테이블 조회, Then 앱에서 생성·수정·삭제한 내용과 일치한다.
- AC-5.2 Given 마이페이지(`/mypage`), When 진입, Then 내가 쓴 글 목록이 표시되고 각 글에서 수정/삭제로 바로 이동할 수 있다.

### FR-6 · 카테고리 · 태그
- US-6.1 사용자로서, 글을 카테고리로 분류하고 태그를 붙이고 싶다.
- US-6.2 사용자로서, 목록에 없는 카테고리는 직접 새로 만들어 쓰고 싶다.

- AC-6.1 Given 글쓰기 폼, When 기존 카테고리 선택, Then `posts.category_id`에 저장된다.
- AC-6.2 Given 글쓰기 폼, When 태그 여러 개 입력, Then `post_tags`에 N:M으로 저장된다.
- AC-6.3 Given 글쓰기 폼의 "새 카테고리" 입력란에 목록에 없는 이름 입력, When 제출, Then `categories`에 새 행이 생성되고 그 카테고리로 저장된다(기본 시드: 바이브코딩·일기·식단·건강·운동).

## 5. 비기능 요구사항 (NFR)

| 구분 | 요구사항 | 검증 관점 |
|---|---|---|
| 보안 | 비밀번호는 Supabase Auth가 자동으로 해시 저장, 평문 저장 없음 | Supabase 대시보드 Authentication 메뉴에서 비밀번호 원문이 노출되지 않는지 확인 |
| 인가(권한) | 본인 글만 수정·삭제 가능, DB의 RLS 정책(`auth.uid()`)에서 강제 | 타 계정으로 접근·API 직접 호출 시도 시 차단 확인 |
| 환경설정 | Supabase 키는 `.env`에 보관, 코드/GitHub에 노출 금지 | 저장소에 키가 하드코딩되어 있지 않은지 확인 |
| 로컬 우선 검증 | 배포 전 Live Server로 로컬 동작을 먼저 확인 | localhost 구동 화면으로 확인 |
| 호환성 | 최신 데스크톱 브라우저 기준 | Chrome 등에서 핵심 흐름(가입→작성→수정→삭제) 수행 |

## 8. 의존성 & 오픈 이슈

**확정됨**: ① DB는 Supabase(PostgreSQL) 사용 ② 프론트는 정적 HTML + Live Server 기준(프레임워크 도입은 이번 범위 아님) ③ 인증은 Supabase Auth(이메일/비밀번호) 사용 — 커스텀 `users.password` 테이블 대신 `auth.users` + `profiles` 확장 테이블 구조로 전환 (근거: 커스텀 테이블 + anon key로는 "본인 글만 수정" 인가를 DB 단에서 강제할 수 없음 — `crud-pure-clarke.md` 1번 ERD 참고)

**오픈 이슈**:
- 이미지 스토리지를 Supabase Storage로 할지 외부 서비스로 할지 — 구현 단계에서 결정
- `category_id`의 NULL 허용(미분류) 여부 — `crud-pure-clarke.md` 3번 고려사항 참고해 결정
- 로그인 상태 유지 방식(세션 vs 토큰) — 구현 단계에서 확정

완료 후 검증은 별도 문서 `checklist.md` 참고 (본 문서 "2. 목표 & 성공지표(KPI)" 기준으로 작성).
